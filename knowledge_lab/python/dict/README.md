# dict — Python 字典

## 一句话概括

`dict` 是 Python 内置的键值对（key-value）容器，底层实现为哈希表。它是 Python 中最常用的数据结构之一，用于存储具有映射关系的数据。自 Python 3.7 起，字典保证**插入顺序有序**。

简单来说：一个可以通过"名字"快速找到"值"的存储结构。

## 常用操作速查

| 操作 | 代码 | 说明 |
|------|------|------|
| 创建 | `d = {"a": 1, "b": 2}` | 花括号字面量 |
| 创建 | `d = dict(a=1, b=2)` | 关键字参数构造 |
| 创建 | `d = dict([("a", 1), ("b", 2)])` | 可迭代对象构造 |
| 取值 | `d["a"]` | key 不存在时抛出 `KeyError` |
| 安全取值 | `d.get("a")` | key 不存在返回 `None` |
| 安全取值 | `d.get("a", 0)` | key 不存在返回默认值 `0` |
| 设默认值 | `d.setdefault("a", 0)` | key 不存在则设值并返回，存在则返回原值 |
| 新增/修改 | `d["c"] = 3` | 存在则覆盖，不存在则新增 |
| 删除 | `del d["a"]` | key 不存在抛出 `KeyError` |
| 删除并返回 | `d.pop("a")` | key 不存在抛出 `KeyError` |
| 删除并返回 | `d.pop("a", None)` | key 不存在返回默认值 |
| 弹出最后项 | `d.popitem()` | 3.7+ 弹出最后插入的项 (LIFO) |
| 检查存在 | `"a" in d` | 检查 key 是否存在 |
| 合并 | `d1 \| d2` | 3.9+，后者覆盖前者 |
| 原地合并 | `d1 \|= d2` | 3.9+，后者覆盖前者 |
| 更新 | `d.update(d2)` | 将 d2 的键值对写入 d |
| 解包合并 | `{**d1, **d2}` | 3.5+，后者覆盖前者 |
| 遍历 key | `for k in d:` | 等同于 `for k in d.keys()` |
| 遍历 value | `for v in d.values():` | 返回值的视图 |
| 遍历键值 | `for k, v in d.items():` | 返回 (key, value) 的视图 |
| 字典推导 | `{k: v*2 for k, v in d.items()}` | 创建新字典 |
| 长度 | `len(d)` | 键值对数量 |
| 清空 | `d.clear()` | 清空所有键值对 |

## 关键特性

- **Key 必须可哈希**：`str`、`int`、`float`、`tuple` 可作为 key；`list`、`set`、`dict` 不可作为 key。
- **插入有序**（3.7+）：字典保持键值对的插入顺序，遍历顺序与插入顺序一致。
- **动态扩容**：字典会根据元素数量自动扩容，开发者无需关心容量。
- **查找 O(1)**：平均情况下，查找、插入、删除的时间复杂度均为 O(1)。

## collections 扩展

| 类 | 说明 |
|----|------|
| `defaultdict` | 访问不存在的 key 时自动生成默认值 |
| `OrderedDict` | 有序字典（3.7 前有用，3.7 后普通 dict 已有序） |
| `Counter` | 计数器，统计元素出现次数 |
| `ChainMap` | 将多个字典视为一个整体进行查找 |

## 演进历史

| 版本 | 年份 | 变化 |
|------|------|------|
| Python 0.9 | 1991 | Python 诞生时就有了字典类型，基于哈希表实现，无序 |
| Python 2.2 | 2001 | 字典的 `pop`、`setdefault`、`fromkeys` 方法加入 |
| Python 2.3 | 2003 | `collections` 模块引入，包含 `deque`、`defaultdict` 等 |
| Python 2.7 | 2010 | `collections.OrderedDict` 加入——一个能记住插入顺序的字典（内部维护双向链表） |
| Python 3.0 | 2008 | `dict.keys()` / `values()` / `items()` 不再返回列表，改为返回**视图**（view），更节省内存；`has_key()` 方法移除，统一使用 `in` |
| Python 3.5 | 2015 | `{**d1, **d2}` 字典解包合并写法被正式接受（PEP 448） |
| Python 3.6 | 2016 | CPython 实现 "compact dict"（Raymond Hettinger 设计），作为实现细节字典**变有序**了，且内存占用减少 20-25% |
| Python 3.7 | 2018 | 字典插入顺序保留成为**语言规范**（不再只是 CPython 的实现细节），所有 Python 实现都必须遵守 |
| Python 3.9 | 2020 | 新增 `|` 合并和 `|=` 更新运算符（PEP 584），字典合并有了更直观的语法 |
| Python 3.10 | 2021 | `dict.keys()` / `values()` / `items()` 视图新增 `mapping` 属性，可直接访问回原字典 |
| Python 3.11 | 2022 | 字典核心代码进一步优化，`dict.popitem()` 行为修正为 LIFO |

> **Python 2 → 3 迁移注意**：Python 2 中 `d.keys()` 返回列表（复制一份），Python 3 中返回视图（实时反映字典变化）。遍历时修改字典在 Python 3 的视图中会抛出 `RuntimeError`。

## 历史故事：从哈希表到 compact dict

字典的底层是哈希表——这个数据结构的历史可以追溯到 1953 年，IBM 的程序员们需要在汇编语言里快速查找符号名。哈希表的核心思想很简单：给定一个 key，通过哈希函数算出它在数组里的位置，直接读取。理想情况下，一次就能找到。

但现实中，不同的 key 可能会"撞"到同一个位置，这叫**哈希冲突**。解决冲突的方式各有流派：有的用链表把冲突项串起来（拉链法），有的去找下一个空位（开放寻址法）。

Python 从诞生起就使用开放寻址法。一直用到 Python 3.5 的字典实现中，每个键值对占据一个"entry"，里面存着 key、value 和 hash 值，表里还会留下大量"墓碑"（已删除项的占位标记）。这种设计有三个问题：内存占用大（entries 稀疏分布）、无序（遍历顺序取决于哈希值）、删除后的墓碑会拖慢查找。

2012 年，Python 核心开发者 **Raymond Hettinger** 提出了一个大胆的想法：为什么不把 key-value 拆成两张表？一张是索引表（indices），只存 entry 的序号；另一张是 entries 稠密数组，按插入顺序排列 key-value 数据。这样——

- **entries 按插入顺序存储**，字典自然有序，不再需要 `OrderedDict` 的双向链表。
- **索引表只用 1 字节**（小字典）或更多（大字典），大幅节省内存。
- **遍历效率极快**，直接在稠密 entries 数组上迭代，不跳空位。

这个设计在 Python 3.6 被合入 CPython，实测内存占用降低了 20-25%，遍历速度提升了近一倍。因为效果太好，2018 年到来的 Python 3.7 把"字典有序"写进了语言规范——从此所有 Python 实现都必须保证字典按插入顺序排列。

2020 年 Python 3.9 带来了 `|` 运算符，两个字典的合并终于不需要 `{**a, **b}` 这种间接写法。这一路，从 1953 年的哈希表论文，到 Python 0.9 的第一个字典实现，到 2016 年的 compact dict 重构，再到 2020 年的 `|` 运算符——小小的字典，凝聚了六十多年数据结构研究的精华。

你代码里的那行 `d = {"name": "张三", "age": 25}`，底层是一张精心设计的哈希表，键的插入顺序被忠实保留，查找在眨眼之间完成——就像 1953 年的 IBM 程序员幻想过的那样。
